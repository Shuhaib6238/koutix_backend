Architecture Overview

The Koutix POS Connection system is a robust integration framework that synchronizes point-of-sale data from multiple POS vendors (LS Retail, SAP) into the Koutix platform. The system supports two synchronization methods:

  1. API Pull - Koutix periodically pulls data from the POS API
  2. Webhook - POS system pushes data directly to Koutix

The architecture ensures:

  • Secure credential storage (AES encryption)
  • Flexible authentication methods
  • Reliable data synchronization
  • Complete audit trail of all sync events
  • Real-time status monitoring

Data Model

Store.posConnection Schema

{
  posType: String,              // 'ls_retail' | 'sap' | 'custom' | null
  method: String,               // 'api_pull' | 'webhook' | null
  status: String,               // 'disconnected' | 'connected' | 'error'
  encryptedCredentials: String, // AES-256 encrypted
  webhookSecret: String,        // 32-byte hex for validation
  pullIntervalSeconds: Number,  // default: 300 for api_pull
  lastSyncAt: Date,             // last sync timestamp
  lastSyncStatus: String,       // 'success' | 'fail' | null
  lastErrorMessage: String      // error details
}

Connection Flow

Phase 1: Validation

  1. Verify POS type is valid (ls_retail, sap, custom)
  2. Verify sync method is valid (api_pull, webhook)
  3. Verify credentials object is present and valid

Phase 2: Authentication

  1. Verify user is authenticated
  2. Verify user has branch_manager role or higher
  3. Verify user has access to the specified store

Phase 3: Connection Testing

For api_pull:

  • Make real API call to POS endpoint
  • Verify authentication works
  • Verify API returns valid response

For webhook:

  • If sapServerUrl provided, send test ping
  • Verify server responds
  • If no URL, assume ready to receive

Phase 4: Credential Encryption

  • Serialize credentials to JSON
  • Generate random AES-256 IV
  • Encrypt using AES-256-CBC
  • Store encrypted blob in database

Phase 5: Webhook Secret Generation

  • Generate 32 random bytes
  • Convert to hex string (64 characters)
  • Store in database
  • Return to client (only at connection time)

Phase 6: Job Scheduling

If api_pull: Create BullMQ repeating job every pullIntervalSeconds

If webhook: No job needed, wait for incoming requests

API Pull Implementation

Process Flow

  1. Decrypt credentials from database
  2. Calculate time window (lastSyncAt or 24h ago)
  3. Build authentication header (Bearer token or Basic auth)
  4. Make API request to POS endpoint
  5. Parse response (handle array or object formats)
  6. Convert using adapter to Koutix format
  7. Log all events (success and errors)
  8. Update store record with sync status
  9. Log summary

Error Handling

Decryption Error: Log error, mark sync as failed
API Call Error: Log error, update lastErrorMessage
Conversion Error: Log specific error, continue processing

Webhook Implementation

Endpoint: POST /api/pos/webhook/:branchId/:posType

Security Features:

  • No authentication required (secret-based validation)
  • Validates x-webhook-secret header
  • Validates POS type matches
  • Validates connection status is 'connected'

Response Strategy

Always returns 200 OK with message "Webhook received"

This prevents the POS system from retrying. Processing happens asynchronously after response is sent.

Processing Steps

  1. Normalize payload to array
  2. Convert with appropriate adapter
  3. Log successful and failed conversions
  4. Update store sync status
  5. Log any errors (but don't retry)

Security & Encryption

Credential Encryption

  • Algorithm: AES-256-CBC
  • Key Size: 256 bits
  • IV: Random 16 bytes per encryption
  • Format: iv:encryptedData (hex:hex)

Webhook Secret

  • Generated: 32 random bytes → 64 hex characters
  • Stored in: store.posConnection.webhookSecret
  • Validated via: x-webhook-secret header
  • Unique per connection

Credential Protection

  • Never returned in API responses
  • Only decrypted when needed for API calls
  • Never logged in plaintext
  • Errors sanitize sensitive data

Event Logging

PosEvent Model

{
  branchId: ObjectId,
  posType: String,
  rawPayload: Object,        // Original from POS
  convertedPayload: Object,  // Standardized format
  status: 'success' | 'fail',
  errorMessage: String,      // If status === 'fail'
  receivedAt: Date,
  timestamps: { createdAt, updatedAt }
}

Event States

Successful Sync:

  • rawPayload: transaction data
  • convertedPayload: standardized format
  • status: 'success'
  • errorMessage: null

Failed Sync:

  • rawPayload: transaction data
  • convertedPayload: null
  • status: 'fail'
  • errorMessage: reason for failure

Adapter System

The adapter pattern provides flexible handling of different POS formats:

POS System → Raw Data → Adapter → Standardized Format → Koutix

Standardized Event Format

{
  transactionId: String,
  amount: Number,
  currency: String,
  timestamp: String,        // ISO 8601
  items: Array,
  customer: {
    id: String,
    name: String,
    phone: String
  },
  metadata: Object
}

Error Handling

Error Status Code Message
Invalid POS type 400 Invalid posType. Must be one of: ls_retail, sap, custom
Invalid method 400 Invalid method. Must be "api_pull" or "webhook"
No credentials 400 Credentials are required
Store not found 404 No store found for this branch manager
Connection test failed 400 Connection test failed: {error message}
Invalid webhook secret 401 Invalid webhook secret

Key Features

  • Encrypted credentials - Never stored in plain text
  • Webhook secret validation - Prevents unauthorized webhooks
  • Connection testing - Validates before saving
  • Async processing - Webhooks return immediately
  • Event audit trail - All sync attempts logged
  • Error tracking - Last error persisted for UI
  • Flexible auth - API keys, basic auth, or webhook secrets
  • Multiple POS support - Extensible adapter architecture
